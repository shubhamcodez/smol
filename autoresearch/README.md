# Local autoresearch loop

This is a small, runnable adaptation of Andrej Karpathy's autoresearch pattern:
a fixed evaluator, one comparable metric, equal wall-clock experiments, a tiny
mutable surface, an append-only ledger, and keep/discard decisions. It adds a
deployment gate so an accuracy improvement is rejected if it cannot run on the
target accelerator.

The local Snapdragon X Elite path is intentionally split:

- training: NumPy on CPU, because Qualcomm QNN/Hexagon is inference-only;
- deployment: static uint8/int8 QDQ ONNX on the Hexagon NPU through the QNN HTP
  backend, with CPU execution-provider fallback disabled.

On an NVIDIA computer, installing CUDA-enabled PyTorch makes the same benchmark
choose CUDA for both training and deployment timing.

## Snapdragon NPU

Use a native ARM64 Python. An x64 Python running under emulation cannot load the
ARM64 QNN plugin.

```powershell
cd D:\AGI
.\.venv-arm64\Scripts\python.exe .\autoresearch\npu_smoke.py
.\.venv-arm64\Scripts\python.exe .\autoresearch\loop.py --reset --iterations 4 --budget-seconds 1 --deployment qnn-npu
.\.venv-arm64\Scripts\python.exe .\autoresearch\audit.py --expected-deployment qnn-npu
```

The smoke test writes `artifacts/npu_smoke/npu_proof.json`. The proof records an
NPU-type `QNNExecutionProvider` device, the HTP backend path, disabled CPU EP
fallback, model checksum, and measured latency. The research ledger is
`results.tsv` and every trial has its own model and `metrics.json` artifact.

## NVIDIA CUDA computer

Use 64-bit Python 3.11 or 3.12 and install a CUDA-enabled PyTorch build suited to
that machine, then:

```powershell
python -m pip install -r .\autoresearch\requirements-cuda.txt
python .\autoresearch\nvidia_smoke.py
python .\autoresearch\loop.py --reset --iterations 8 --budget-seconds 5 --training-backend cuda --deployment cuda
```

`nvidia_smoke.py` performs a real forward, loss, backward, and optimizer step on
a scaled instance of `model.py`, and asserts that its logits are CUDA tensors.
The default model is 203.8M parameters with a 4,096-token context, sized for
mixed-precision, checkpointed training with micro-batch size 1 on a 6GB GPU.
Use the command generated by the official [PyTorch installation
selector](https://pytorch.org/get-started/locally/) when the CUDA wheel required
by that NVIDIA driver differs from the ordinary requirements file.

## What self-improves

The built-in loop hill-climbs hidden width, learning rate, weight decay, batch
size, and Adam beta settings. A trial is accepted only when the combined score
improves. The fixed score is validation cross-entropy plus small latency and
parameter-count terms. This is a proof-grade local harness, not a claim that a
small proxy task automatically discovers frontier language-model algorithms.
For language-model research, retain the orchestrator and deployment gate, then
replace only the fixed task/model pair with a real tokenized training shard and
held-out bits-per-byte evaluator.

Design references: [Karpathy autoresearch](https://github.com/karpathy/autoresearch),
[Windows ML execution providers](https://learn.microsoft.com/windows/ai/new-windows-ml/supported-execution-providers),
and [ONNX Runtime QNN EP](https://onnxruntime.ai/docs/execution-providers/QNN-ExecutionProvider.html).
